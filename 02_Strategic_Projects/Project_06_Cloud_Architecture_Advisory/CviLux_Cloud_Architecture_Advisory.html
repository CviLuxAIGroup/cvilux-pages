EXECUTIVE SUMMARY

結論摘要

先看建議結論；細節在後續章節展開

議題 1建議：採用

鼎新 ERP + M365 → Azure Fabric Data Lake

合理選擇，與既有 Azure AD / D365 / SharePoint 生態一致；補充 3 個必補設計：Medallion 架構、跨境分區、鼎新連接器評估。

議題 2建議：Azure 為主

CviCloud + 騰雲合作 → 雙雲評估

強烈建議 Azure 主雲（IoT 服務齊全、大陸合規、與集團一致）；Google Vertex AI 作為 LLM 推理輔助。⚠️ Google Cloud IoT Core 已退役是關鍵負面信號。

商模衝擊需董事會討論

騰雲合作對 CviCloud 商業模式的影響

瀚雲變成「硬體 + IoT 整合服務商」而非真正 SaaS 業者；CviCloud 對外宣傳之「自有 SaaS 平台」定位需重新檢視；分潤模式、客戶 NDA、DPA 資料處理協議須法務簽訂。

01

議題 1：Azure Fabric

鼎新 ERP + M365 → Fabric Data Lake

將鼎新 ERP 與 Microsoft 系統資料整合至 Azure Fabric Data Lake — 此為合理選擇，但需補強 3 個關鍵設計。

✅ 為什麼合理 — 5 個支持理由
1. 生態一致

已採 Azure AD / D365 / SharePoint，Fabric OneLake 對 M365 為原生整合，無需重複連接器

2. Power BI 內建

Fabric 直接提供 Power BI Premium 容量，集團報表無縫

3. AI 整合

與 Azure ML、Azure OpenAI、M365 Copilot 一條龍；與您先前選定的 Multi-Agent 架構一致

4. Lakehouse 統一

結構化（ERP）+ 非結構化（PDF / 圖片 / 影音）統一管理

5. 持續發展

Fabric 為 Microsoft 戰略產品，每季功能擴張 推論

6. 大陸合規

Azure 在大陸有 21Vianet 獨立部署，符合 PIPL 跨境分區需求

必補 1

Medallion 架構

Bronze / Silver / Gold

不要把 ERP 原表直接拿來分析。設計三層：

  • · Bronze：ERP / MES / SharePoint 原始資料 raw copy
  • · Silver：清洗、去重、Schema 統一
  • · Gold：業務 KPI、跨系統 join 後的分析表

效益：可追溯、可重算、AI 訓練資料品質保證

必補 2

跨境資料分區

PIPL / GDPR / 個資法

**不可**單一 Fabric Workspace 容納全集團資料：

  • · 台灣 Workspace：台灣公司、海外（美 / 荷 / 寮）
  • · 大陸 Workspace：陸區子公司（用 21Vianet 或鼎新雲）
  • · 跨境僅交換 aggregated 統計（如月度產能、成本指標）

效益：合規 + 風險隔離 + 各區優化模型

必補 3

鼎新 ERP 連接器

確認可行路徑

[推論] Fabric 對鼎新 ERP 沒有原生連接器；可能路徑：

  • · 鼎新 WF AIGP 升級後是否有原生 Fabric Sink？
  • · 用 Azure Data Factory + ODBC 連 ERP 資料庫
  • · 用 Logic Apps + ERP REST API（增量）
  • · 鼎新雲（如已採購）對 Fabric 的整合方案

建議 IT 副總與鼎新確認後選定主路徑

⚠️ 容量規劃提醒　Capacity Planning

Fabric 採 F2-F2048 容量計費（小時計）；建議從 F32 / F64 起步測試 推論，再依實際負載調整。**Capacity 可暫停**降低週末閒置成本。月度成本估算 NT$ 30-150 萬不等，視容量等級。

03

關鍵事實

3 Critical Facts to Know

下列三個事實會強烈影響選型決策，董事會與 IT 副總須優先了解。

事實 1：Google Cloud IoT Core 已退役 已驗證

Google 於 2022 年 8 月宣布、2023 年 8 月 16 日正式關閉 IoT Core 服務。原 IoT Core 用戶必須遷移至第三方平台（如 ClearBlade IoT Core、HiveMQ）或自建 MQTT broker。

**對 CviCloud 影響**：若選 GCP 為主雲，sCMS 智碳雲（IoT 主導）需自建 IoT 基礎設施或接受第三方廠商，增加複雜度與維運成本；Azure IoT 服務在當前持續投資，是更穩定選擇。

資料來源：Google Cloud 官方退役公告（公開資訊）

事實 2：Azure Fabric 為 Microsoft 戰略產品 推論

Fabric 於 2023 年 5 月發表、2023 年 11 月一般可用（GA）；整合 Power BI、Synapse、Data Factory、Real-Time Analytics、AI 服務於 OneLake。

**選用考量**：作為較新平台，部分鼎新 ERP / MES 連接器可能需自行開發；建議於 PoC 階段（非全集團 Roll-out）先驗證關鍵 Pipeline，避免資源綁死。

事實 3：雙雲管理成本疊加 推論

業界普遍經驗 [推論]：同時維運 2 個雲端供應商之管理成本約為「主雲 + 30-50%」（含人才稀缺、跨雲監控工具、API 同步、資料聯邦等）。除非有強烈業務理由，建議「**單一主雲 + LLM API 級別呼叫第二雲**」之輕量混合，而非真正雙雲對等。

**對應建議**：Azure 為主、Vertex AI 透過 API 呼叫（不部署資料服務在 GCP）—管理成本疊加可控制在 +5-10%。

跨境分區設計

Cross-Border Data Zoning

瀚荃集團跨台 / 陸 / 海外運營，跨境資料是「最大法律風險」。本章建議分區設計。

DATA ZONING ARCHITECTURE
ZONE 1 · 台灣 / 海外

Azure（東亞）

  • · 瀚荃股份 / 瀚雲 / 瀚柔總部
  • · CviLux USA / Opro9 EU
  • · Fabric Workspace（台）
  • · D365 / SharePoint 主資料
  • · 客戶 NDA 文件

合規：個資法 + GDPR

ZONE 2 · 大陸

Azure 21Vianet 或鼎新雲

  • · 東莞 / 蘇州 / 重慶 / 深圳 / 安徽 / 上海
  • · Fabric Workspace（陸）獨立
  • · 鼎新 ERP 大陸實例
  • · 陸區客戶 / 經銷商資料
  • · 微信 / 企業微信整合

合規：PIPL + 數據安全法

ZONE 3 · 跨境聯邦

僅 aggregated 統計

  • · 集團月度 / 季度報表
  • · 全集團 KPI 儀表板
  • · 集團 ESG 報告數據
  • · 客戶 PII 不傳遞
  • · 商業機密不跨境

機制：Power BI 跨域連線 / API

⚠️ 重要：陸區資料 → 台灣 Power BI 不可直接連接明細資料；需先在陸區彙總後傳送 aggregated 結果。建議由法務 + 資安 + IT 三方建立「**跨境資料傳輸白名單**」，明確列出可跨境的資料類型與粒度。
05

商業模式衝擊

Impact on CviCloud Business Model

騰雲合作對 CviCloud 既有商業模式定位之影響 — 需董事會與法務介入決策。

v1.2 規範下的 CviCloud 定位

瀚雲科技股份有限公司 CviCloud Technology Corporation 為瀚荃集團前店子公司，三大商模：(1) 硬體賣斷 + (2) 軟體 SaaS 訂閱 + (3) HaaS 租賃。

⚠️ 衝擊 1：CviCloud 從「SaaS 業者」變「整合服務商」

如騰雲負責軟體維運，CviCloud 實質角色：

  • ✗ SaaS 平台供應商
  • ✓ 硬體 + IoT 整合服務商
  • ✓ 系統整合代理（轉售騰雲 SaaS）

**對外溝通需注意**：是否仍可宣傳「自有 SaaS 平台」？

⚠️ 衝擊 2：客戶資料歸屬與責任

客戶 ESG / CBAM 資料由騰雲管理時：

  • · 客戶 NDA 是否准許第三方存取？
  • · 法律責任如何分攤？
  • · 資料外洩賠償歸誰？
  • · 客戶終止合約後，資料歸屬？

**必簽**：DPA 資料處理協議 + SLA + 退出機制

⚠️ 衝擊 3：收入分潤模式

原 CviCloud 三大商模收入結構需重設計：

  • · 硬體賣斷 — 全歸瀚雲
  • · SaaS — 瀚雲與騰雲分潤（X% 給瀚雲？）
  • · HaaS — 硬體租賃 + 軟體訂閱組合，分潤更複雜
  • · 客服收入 — 一線騰雲？二線瀚雲？

**必設計**：SLA 連動分潤、長期客戶折扣、續約機制

⚠️ 衝擊 4：CviCloud 對集團 AI 平台之資料貢獻

如客戶 ESG 資料在騰雲：

  • · 集團 AI 平台無法直接訓練模型
  • · 集團 ESG Agent 資料源受限
  • · EAP ESG AI Platform 戰略價值降低
  • · sCMS 跨客戶 benchmark 受阻

**對應**：合約中保留瀚雲對 aggregated 統計資料的存取權

RECOMMENDATION · 商業模式調整建議

建議先**確認騰雲合作協議性質**：是「軟體授權代理」還是「SaaS 共同營運」還是「外包技術服務」？三種模式對 CviCloud 商業地位影響截然不同。

  • · **代理**：騰雲為瀚雲授權 SaaS，瀚雲對外仍為單一供應商 — 衝擊小
  • · **共同營運**：雙品牌雙簽約，分潤明確 — 衝擊中
  • · **技術外包**：騰雲純粹 ODM 後台維運，瀚雲為唯一面向客戶 — 衝擊小

**建議：「技術外包」模式 + Azure 主雲，最能保留 CviCloud 之 SaaS 業者品牌定位。**
