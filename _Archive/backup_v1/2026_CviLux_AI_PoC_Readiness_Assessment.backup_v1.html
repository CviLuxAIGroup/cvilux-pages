01

PoC #1

安全庫存預測智能助手

Smart Safety Inventory Forecasting Assistant

業務問題　Business Problem

集團 14+ 廠跨地理（淡水 / 東莞 / 蘇州 / 重慶 / 越南 / 寮國 / 美國）面對需求波動 + 客戶等級報價 + 長前置時間零件採購；目前安全庫存以人工經驗法則設定，導致：(1) 部分料號缺料急採高溢價；(2) 部分料號呆滯佔資金。

PoC 目標　PoC Goal

在 1 個試點廠（建議東莞） 對 top 100 SKU（依 Pareto，覆蓋 80% 庫存價值）建立 AI 預測模型，3 個月 Shadow Mode 比對 AI 建議 vs 採購人員實際決策，KPI 達標後擴展至全廠 / 全集團。

需 36 個月以上 歷史資料；每筆需有時間戳記、廠別、料號、單位；資料完整性 ≥ 95%。

來源系統資料表 / 欄位用途必要性取得難度

鼎新 ERP採購單歷史（PO 日期、料號、數量、實到日、供應商、單價）前置時間 + 供應端波動必易
鼎新 ERP庫存異動日表（日期、料號、廠別、入庫、出庫、結存、呆滯標記）時序主資料必易
鼎新 ERP銷售訂單 / 出貨記錄（客戶、料號、數量、實際出貨日）需求預測訓練必易
MES實際領料記錄（工單、料號、領用量、領用時間）產線消耗速率必中
MES不良品 / 報廢數量耗用率修正應中
D365 CRM訂單預測 / Pipeline / 客戶等級未來需求訊號應中
SharePoint採購合約（最低訂購量 MOQ、最小批量、價格階梯）規則約束應中
外部原物料市場價格（銅、鋁、塑料）成本與替代評估可難
外部節氣 / 節假日 / 物流中斷新聞季節性 + 風險可難
內部知識採購人員經驗規則（已知 ROP、EOQ、安全係數）RAG 知識庫，避免冷啟動必中（需訪談）

DIMENSION 4

AI 模型選型　Model Selection

採用 3 層混合架構：時序預測（量）+ 規則引擎（約束）+ LLM 解釋（建議）。

層模型輸出優點考量
L1 量預測Prophet（基線）每日 / 每週需求預測季節性強、易解釋對突發事件不敏感
L1 量預測LightGBM短期需求（1-4 週）處理特徵多、訓練快需特徵工程
L1 量預測LSTM / TFT長期需求（4-12 週）學長序列依賴需大量資料、訓練慢
L2 安全水位規則引擎 + 凱利公式 / Newsvendor動態 ROP / EOQ可解釋、可調需專家規則
L3 解釋層LLM (Gemini / GPT-4) + RAG「為什麼建議調高 / 調低」提升信任、便於審核需控幻覺
PoC 階段建議： Prophet + LightGBM + 規則引擎 + LLM 解釋。後期再評估 Transformer / TFT。
輸出格式： 每料號每日：建議安全水位、預測 4 週需求、95% 信心區間、自然語言解釋、3 個替代料推薦。

02

PoC #2

生產線效率優化助手

Production Line Efficiency & Cost Optimization Assistant

業務問題　Business Problem

FFC 線「200+ 潔淨室等級設備」、SMT、Connector 注塑、組包等複雜製程；目前以人工 IE 工程師 + Excel 分析瓶頸；缺乏即時 OEE（可用率 / 性能 / 品質）視覺化、缺乏跨班 / 跨機台統一比較、新人接手知識傳承困難。

PoC 目標　PoC Goal

在 1 條試點線（建議東莞 FFC 高頻線） 部署 IoT + Process Mining + AI 助手，3 個月 找出 Top 5 瓶頸與改善建議，目標 OEE +5 個百分點、單位能耗 -8%。

與 PoC #1 的關鍵差異

PoC #1 主要是「歷史資料 + 預測」，PoC #2 是「即時資料 + 行動建議」。後者需要邊緣運算、IoT 閘道、時序資料庫，技術投入比 PoC #1 高 30-50%，但對製造現場影響更直接。

兩個 PoC 共用同一套治理機制：資料治理、AI 治理、變更管理、PoC → 量產化路徑。本框架由永續聯辦公室統籌、IT + 法務 + 資安 + AI 治理委員會共同審議。

資料治理 Data Governance

  • · 每張資料表設 Owner
  • · 資料字典強制版本控管
  • · 品質閾值 + 月度報告
  • · 個資 / 商秘遮罩 SOP
  • · 跨境資料分區（PIPL）

AI 治理 AI Governance

  • · Human-in-the-loop 強制
  • · Model Card 必交付
  • · Audit Log 7 年保留
  • · Bias / Drift 月度檢測
  • · LLM 來源引用率 100%

變更管理 Change Mgmt

  • · 範圍變更需 Sponsor 簽核
  • · 模型升級走 CI/CD + Canary
  • · 規則修改業務 + IT 雙簽
  • · 月度進度董事會匯報

量產化 Productionization

  • · PoC 通過後 6 個月內擴展
  • · SLA 必達 99.5%
  • · 維運責任移轉給內部
  • · 持續模型再訓練（季度）
POC → PRODUCTION PATHWAY · 量產化路徑

從 PoC 到全集團部署的 4 個階段

Stage 1 · PoC（3 個月）

單一試點（廠 / 線）+ Top 100 對象。預算 NT$ 500-850 萬 / 個

Stage 2 · Pilot Plus（3 個月）

擴展至全試點廠所有 SKU / 全產線 OEE。預算 NT$ 800-1,500 萬

Stage 3 · 集團 Roll-out（6-9 個月）

擴展到 4 大廠（淡水 + 東莞 + 蘇州 + 越南）。預算 NT$ 2,500-4,000 萬

Stage 4 · 全集團（12 個月）

14+ 公司全覆蓋 + 持續優化。年度維運 NT$ 800-1,200 萬

關鍵 Gate：每階段結束 KPI 報告 → 董事會核定下階段預算與範圍；任一階段未達標可暫停 / 終止 / 重設範圍。

啟動會議決策清單

14 Critical Decisions

下列 14 條為兩個 PoC 啟動前必須在董事長主持的啟動會議上決議；任一條未決則該 PoC 不得進入 D+30 簽約。

1. 試點廠 / 試點線

PoC #1 是否選東莞？PoC #2 是否選 FFC 高頻線？

2. 試點範圍 SKU / 產線數

PoC #1 是否 Top 100？PoC #2 是否單線？

3. PoC 預算上限

兩 PoC 合計約 NT$ 1,000-1,400 萬，是否核准？

4. 主合作雲廠 / 平台

是否確認以 Azure 為主、Vertex AI 輔助？

5. 採購 / 廠長 owner 人選

兩個 PoC 各自指定 1 名業務主導者

6. 跨境資料策略

大陸資料是否留陸區？只傳 aggregated？

7. AI 治理委員會組成

永續聯辦 + IT + 法務 + 資安 + 業務代表

8. Human-in-the-loop 強制

所有 AI 建議必經人工審核才執行（不直接寫 ERP）

9. KPI 達標標準

兩 PoC 各自的 Go/No-Go 標準是否確認？

10. 知識轉移機制

外部顧問需培訓內部 X 名於 4 週內接手

11. WF AIGP 對接時程

第一階段是否依賴 WF AIGP？或先用 ERP 既有 API？

12. 失敗終止條件

什麼情況可中途終止？預算止損點？

13. 對外宣傳策略

PoC 階段是否對客戶 / 投資人公開？以 ESG 故事？

14. 量產化預先承諾

若 PoC 成功，是否預先承諾 Stage 2 預算？

⚠️ 強烈建議

啟動會議時間 不超過 2 小時；14 條決策以「Yes/No」形式提案，附本評估書作為決策依據；現場無法定案者列「待決事項」，3 個工作日內由 PMO 補資料再次出案。
